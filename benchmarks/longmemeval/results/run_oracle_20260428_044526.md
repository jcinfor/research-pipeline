## Caveat — m_flow's column is a degraded-config measurement, not its capability ceiling

The m_flow paper reports **89% LLM-judge on LongMemEval** (93% temporal, 82% multi-session). We measure 19% / 24% / 2.5% here. The gap is the adapter, not m_flow:

1. **Missing env vars.** m_flow's own `examples/test_comprehensive_episodic.py` enables `MFLOW_EPISODIC_ENABLE_FACET_POINTS=true`, `MFLOW_EPISODIC_POINT_REFINER=true`, `MFLOW_EPISODIC_RETRIEVER_MODE=bundle`. Our adapter set none of these. Without `ENABLE_FACET_POINTS`, m_flow doesn't build the granular layer where literal evidence lives — retrieval has only Episode-level summaries to surface. This is the most likely single cause of the multi-session collapse (1/40).
2. **Wrong synthesis path.** We used `m_flow.search(query_type=EPISODIC)` and ran our own Gemma synthesis on the returned summary text. m_flow's reported numbers use TRIPLET_COMPLETION + bundle synthesis — m_flow's own LLM call with the full bundle (Episode + Facets + FacetPoints) in context. We discarded m_flow's bundle reasoning.
3. **Tool-call instructor mode.** Our local vLLM doesn't have `--guided-decoding-backend` configured, so json_schema-mode requests hang. We forced `MFLOW_LLM_INSTRUCTOR_MODE=tool_call` to get past the hang. m_flow's extraction prompts are tuned for json_schema_mode; tool_call mode produces coarser/different extractions during ingest.
4. **Local Gemma 4 26B vs the paper's frontier LLM** (m_flow defaults to `gpt-5-nano`). For sparse-context recall, the model gap matters.
5. **Cross-event-loop errors** — 6/100 m_flow jobs returned `(mflow retrieval error: <asyncio.locks.Lock ...>)` because m_flow caches module-level Locks bound to whichever instance's loop first touched them. Our per-instance-loop fix didn't catch all cases. These are counted as judge=✗ but are infrastructure failures, not retrieval misses. Without them m_flow's effective rate is ~21% rather than 19%.

A focused **m_flow-only re-run with proper config** (the four `MFLOW_EPISODIC_*` env vars, TRIPLET_COMPLETION + use_combined_context, single-thread to sidestep #5) is in flight and tracked separately. Treat the 19% number above as the lower bound for m_flow against this hardware/LLM stack, not a measurement of m_flow's algorithmic ability.

The other three columns (prototype 56%, multitier 53%, mem0_real 53%) are configured the same way they've always been and are directly comparable to Phase B numbers and to each other.

## Operational note — m_flow-only runs are slower than multi-system runs (counterintuitive)

The proper-config m_flow-only re-run launched after this Phase C is paradoxically *slower* per question than m_flow was inside Phase C. Reason: the `MFlowReal` adapter routes all calls through a single shared asyncio event loop on a dedicated thread (necessary to avoid m_flow's module-level cached Locks binding to the wrong loop and crashing). Inside that loop, async LLM calls parallelize fine, but **kuzu graph queries and lancedb vector-store ops are sync** — they serialize.

- **Phase C (4 systems, 4 workers):** m_flow effectively had ~1 worker active at a time while the other workers powered through the faster systems. No cross-worker contention on m_flow's shared loop.
- **m_flow-only (4 workers, all on m_flow):** all 4 worker threads submit to the same shared loop. Async parts run concurrently; sync graph/vector ops force serialization with extra lock-handoff overhead. Net effective parallelism is ~1, plus contention cost.

Secondary factor: m_flow's `.mflow` store grows globally across all 100 questions (per-question dataset names share the same underlying kuzu DB). Graph-traversal cost is O(graph size), so question 100 touches a much larger graph than question 1. Phase C didn't escape this either but the slowdown was masked by concurrent work on other systems.

**Recommendation for future m_flow-only runs:** pass `--max-workers 1`. Same effective parallelism (loop serializes anyway), no contention overhead, ~20-30% faster wall-clock. Or, if a fully-isolated graph per question is desired, call `m_flow.prune.prune_data()` in `MFlowReal.__init__` — but that forces sequential question processing, which `--max-workers 1` already gives.
